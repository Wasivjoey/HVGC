# 🎚️ HVGC LINEUP

A feature-rich web application for scheduling volunteers to serve on an
**audio / visual team**. It has a **volunteer (user) side** and an
**administrator side**, and is built to run locally or on any cloud platform.

---

## ✨ Features

### Volunteer side
- **Self-service accounts** — register with a username, **verify your email**, and
  sign in with **username or email** once an admin approves you. **Forgot-password**
  reset is built in.
- **Availability** — mark yourself **available or unavailable** for a single day
  **or a whole date range**, with quick-pick for upcoming Sundays.
- **Services** — browse upcoming service times, see the full roster, download the
  **programme flow document**, and **confirm or decline** your assignments.
- **Role swaps** — hand off a slot you can't cover; qualified teammates volunteer
  to pick it up.
- **Service notes** — leave notes on any service time for the whole team.
- **Training** — read the material and watch the **training video** (which auto-
  pauses if you leave the window); complete required training before you serve.
- **Profile** — update contact details and see your role qualifications.

### Administrator side
- **People management** — promote/demote admins, activate/deactivate accounts.
- **Roles** — full create/edit/delete of serving positions (Audio, Lighting,
  Camera, etc.).
- **Role assignment** — assign roles to volunteers; required training is queued
  automatically.
- **Training** — author training (linked to a role, marked required/optional),
  assign it, and mark completion.
- **Services** — create upcoming services and edit their **title, date, start
  time, location, notes**, and **attach a programme flow document** (PDF, Word,
  PowerPoint, etc.) that the team can download.
- **Scheduling board** — per service, assign people to roles. The board shows each
  candidate's **qualification** and **availability**; unqualified people are
  blocked unless you explicitly override.
- **Availability grid** — at-a-glance matrix of who's available for each service.
- **Swap approvals** — approve a swap to automatically reassign the slot.

### The qualification rule
A volunteer can only be scheduled for a role once they have completed **every
required training** linked to that role. This is enforced on the admin scheduling
board (with an override) and on volunteer swap coverage.

## 🚀 Run it locally

```bash
pip install -r requirements.txt
python3 app.py
```

Open **http://localhost:8000**.

**Seeded admin login:** `admin@avteam.app` / `admin123`

> The first person to register on a fresh database also becomes an admin, so the
> app is usable out of the box. The database (`avteam.db`) and default roles,
> trainings, and sample services are created automatically on first run.

## 🚀 Deploy to Render.com (recommended)

A [`render.yaml`](render.yaml) Blueprint is included that provisions a **Postgres
database** and the **web service** together, wired up automatically.

1. Push this project to a GitHub (or GitLab) repository:
   ```bash
   git init && git add -A && git commit -m "HVGC LINEUP"
   git branch -M main
   git remote add origin https://github.com/<you>/<repo>.git
   git push -u origin main
   ```
2. In Render: **New ➜ Blueprint**, select your repo, and click **Apply**. Render
   reads `render.yaml` and creates the database + web service.
3. First deploy creates the schema and a starter admin automatically. Sign in at
   `https://<your-service>.onrender.com` with **`admin` / `admin123`** and change
   the password immediately (Profile, or via Forgot password).

What the Blueprint sets for you:
- **`DATABASE_URL`** — injected from the managed Postgres database.
- **`SECRET_KEY`** — a strong random value generated by Render.
- **`PYTHON_VERSION`**, **`UPLOAD_PATH`**, health checks on `/healthz`, and a
  gunicorn start command.

Notes:
- **Email (optional):** add `SMTP_HOST`, `SMTP_PORT`, `SMTP_USER`, `SMTP_PASSWORD`,
  `SMTP_FROM` in the dashboard to send verification / password-reset emails.
  Without them the app shows those links on screen.
- **Uploaded files:** the free plan has an ephemeral filesystem, so uploaded
  programme documents and avatars are cleared on each deploy. To keep them, attach
  a **disk** (uncomment the `disk:` block in `render.yaml`; requires a paid plan)
  — or upgrade and the data persists at the mounted path.
- The free Postgres and web instances sleep when idle and the database expires
  after the trial window; upgrade the plans in `render.yaml` for production.

## ☁️ Deploy elsewhere (Docker / other PaaS)

The app is a standard WSGI application served by **gunicorn**, so it runs on any
container or PaaS host. A `Dockerfile`, `docker-compose.yml`, and `Procfile` are
included.

### Option A — Docker (works anywhere: AWS, GCP, Azure, Fly.io, Render, Railway…)

```bash
docker compose up --build
# or
docker build -t avteam .
docker run -p 8000:8000 -e SECRET_KEY="$(python3 -c 'import secrets;print(secrets.token_hex(32))')" \
  -v avteam-data:/data avteam
```

### Option B — Platform-as-a-Service (Render / Railway / Heroku-style)
The included `Procfile` is auto-detected:

```
web: gunicorn 'app:create_app()' --bind 0.0.0.0:$PORT --workers 2 --timeout 60
```

Set these environment variables on the platform:

| Variable        | Purpose                                          | Example                          |
|-----------------|--------------------------------------------------|----------------------------------|
| `SECRET_KEY`    | Signs session cookies — **set a long random one**| `openssl rand -hex 32`           |
| `DATABASE_URL`  | **Postgres** connection URL — when set, the app uses Postgres | `postgresql://user:pass@host:5432/db` |
| `DATABASE_PATH` | SQLite file path (used only when `DATABASE_URL` is unset) | `/data/avteam.db`                |
| `UPLOAD_PATH`   | Folder for uploaded programme documents (mounted)| `/data/uploads`                  |
| `PORT`          | Port to bind (most PaaS set this for you)        | `8000`                           |
| `SMTP_HOST`     | Mail server for email verification (optional)    | `smtp.gmail.com`                 |
| `SMTP_PORT`     | Mail server port                                 | `587`                            |
| `SMTP_USER` / `SMTP_PASSWORD` | Mail account credentials           | —                                |
| `SMTP_FROM`     | From address on verification emails              | `no-reply@yourchurch.org`        |

> **Email verification & approval:** new sign-ups must verify their email and then
> be approved by an administrator before they can sign in. If `SMTP_HOST` is unset
> the app runs in "no email" mode — verification links are logged to the server and
> shown on the registration page (and on the admin's People page), so accounts can
> still be verified without a mail server.

> **Database:** the app supports **Postgres** and **SQLite** from the same code,
> selected at runtime:
> - Set **`DATABASE_URL`** to a `postgresql://…` URL and the app uses Postgres
>   (recommended for production and multi-instance scaling). The included
>   `docker compose up` starts a Postgres service and points the app at it
>   automatically. Managed Postgres (RDS, Cloud SQL, Neon, Supabase, Railway,
>   Render) works by setting `DATABASE_URL` to the provider's connection string.
> - Leave `DATABASE_URL` unset and it falls back to **SQLite** at `DATABASE_PATH`
>   — ideal for local development. Put that file on a **mounted volume** so it
>   survives redeploys.
>
> The schema is created automatically on first run on either backend, and a
> built-in migration adds new columns to existing databases.

A `/healthz` endpoint is provided for load-balancer health checks.

## 🗂️ Project structure

```
app.py            # Flask app factory, filters, health check
db.py             # SQLite schema + first-run seed data
helpers.py        # auth guards + qualification logic
auth.py           # register / login / logout
views_user.py     # volunteer-facing routes
views_admin.py    # administrator routes
templates/        # Jinja2 templates (base, auth/, admin/)
static/style.css  # all styling
Dockerfile, docker-compose.yml, Procfile   # deployment
```

## 🔒 Notes on security
- Passwords are hashed with PBKDF2-SHA256 (`werkzeug.security`).
- Sessions are signed cookies (HTTPOnly, SameSite=Lax). **Always set a strong
  `SECRET_KEY` in production.**
- Run behind HTTPS in production (your cloud platform's load balancer typically
  handles TLS termination).
